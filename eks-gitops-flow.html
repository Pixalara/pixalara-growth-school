ClassKira AWS EKS GitOps Deployment Flow

GitHub Actions starts a temporary GitHub-hosted runner. The runner builds/tests the Laravel app, creates the Docker image, authenticates to AWS through OIDC with short-lived credentials, pushes it to ECR, and updates the Kubernetes manifest. Argo CD then syncs the changed manifest to EKS, where Kubernetes performs a rolling update.

1. CI/CD and GitOps Deployment

Image goes to ECR. Manifest change goes to GitHub. Argo CD syncs GitHub to EKS.

GH
GitHub main Developer commits source code to the main branch.
CI
GitHub Actions workflow The main-branch event starts the pipeline defined in `.github/workflows`.
VM
GitHub-hosted runner Temporary Ubuntu VM where build, test, Docker build, tag, and push happen.
composer install php artisan test docker build docker push ECR
ECR
AWS ECR Private Docker image registry. It stores images only; it does not deploy to EKS.
Later: EKS pulls this image from ECR
After image push: workflow commits the new image tag to GitHub, then Argo CD syncs EKS
Git
Kubernetes manifest The workflow commits the new ECR image tag into the manifest file in GitHub.
Argo
Argo CD Runs inside EKS, watches GitHub, detects the manifest update, and syncs it.
EKS
AWS EKS cluster Receives the updated Deployment from Argo CD with the same ECR image tag committed in GitHub. EKS worker nodes pull that exact image tag from ECR before starting the new pods.
Pod
Rolling update New pods pull the new image from ECR, become ready, then old pods are removed.
Who pushes image? The GitHub-hosted runner pushes the Docker image to AWS ECR.
Does ECR push to EKS? No. ECR stores images. After Argo CD applies the updated tag, EKS worker nodes pull that same tag from ECR.
Where does Argo CD work? Inside EKS. It watches GitHub and applies Kubernetes manifests.
What triggers deployment? The Kubernetes manifest image tag commit, not the image push alone.
1. Commit Code lands on GitHub main.
2. Workflow GitHub Actions receives the push event.
3. Runner GitHub starts a temporary Ubuntu runner VM.
4. Build Image Runner tests code and builds the Docker image.
5. Push ECR Runner pushes the tagged image to ECR.
6. Update Git Runner commits the new image tag to the manifest.
7. Argo Sync Argo CD applies the new manifest to EKS.
8. Rollout EKS pulls the same Argo CD-updated image tag from ECR and performs a rolling update.

2. Secure AWS Connection With OIDC

No AWS access key or secret access key is stored in GitHub. The runner receives short-lived AWS credentials only for this job.

VM
GitHub-hosted runner The workflow job runs on a temporary runner and needs AWS access for ECR push.
OIDC
GitHub OIDC token The runner asks GitHub for an identity token proving repo, branch, workflow, and job identity.
IAM
AWS IAM role trust AWS trusts only the allowed GitHub repo and branch, for example `Pixalara/classkira-platform` on `main`.
STS
AWS STS credentials AWS STS exchanges the OIDC identity for temporary AWS credentials scoped to the IAM role.
ECR
Push to ECR The runner uses temporary credentials to login to ECR and push the Docker image tag.
Stored in GitHub No AWS access keys. Use repository variables or workflow env for non-secret values like region and role ARN.
AWS permissions The IAM role should allow only required actions, such as ECR push and read access needed by the job.
Why safer? Credentials are short-lived and issued only when the trusted GitHub workflow runs.
